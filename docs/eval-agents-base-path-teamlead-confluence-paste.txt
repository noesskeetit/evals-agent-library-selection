[Eval Agents]

Решение

Выбираем DeepEval как base path платформы.

OpenEvals не берём как обязательную зависимость. У него более нативный Golden Path API, но это не критично: DeepEval закрывает тот же сценарий через ToolCorrectnessMetric и GEval над serialized trace.

Практический выбор:

    Required: DeepEval
    Optional later: OpenEvals, только если native trajectory API реально начнёт экономить поддержку
    Internal contract: EvalCase, не формат конкретной библиотеки

Почему DeepEval:

- сильнее community;
- шире набор eval-сценариев;
- лучше tracing/OTEL story;
- закрывает оба сценария из ТЗ: Blackbox и Golden Path.

Что тестировали

Агент: LLM-powered погодный агент на moonshotai/Kimi-K2.6 с Open-Meteo tools.

    Task:
    Give me a short weather plan for Moscow, Russia for the next 3 days.

    Actual path:
    geocode_location -> get_weather_forecast -> final answer

Матрица 2x2:

  Variant                              Library     Сценарий                                   Результат
  ------------------------------------ ----------- ---------------------------------------- -----------
  openevals_golden_path_llm_as_judge   OpenEvals   Golden Path по trajectory                       pass
  openevals_blackbox_llm_as_judge      OpenEvals   Blackbox по final answer                        pass
  deepeval_golden_path_llm_as_judge    DeepEval    ToolCorrectnessMetric + GEval по trace          pass

Сравнение по критериям ТЗ

  Критерий                  Вывод
  ------------------------- -----------------------------------------------------------------------------------------------------------------------------------------------------------------------
  Community                 DeepEval заметно сильнее: 16k stars / 1.5k forks против 1k / 98 у OpenEvals.
  Гибкость eval-сценариев   DeepEval шире: RAG, agentic metrics, DAG/custom metrics, multi-turn, MCP, tracing.
  OpenTelemetry / tracing   DeepEval сильнее: есть свой deepeval.tracing и OTEL/Confident AI path. У OpenEvals OTEL-путь идёт через LangSmith ecosystem. Live OTEL export отдельно не поднимался.
  Golden Path ergonomics    OpenEvals удобнее: trajectory для него native input. Но это ergonomics, не blocker: в DeepEval тот же сценарий делается adapter-ом.

Итог по критериям: DeepEval выигрывает как base path. OpenEvals полезен как reference point по API, но не обязателен.

Golden Path, Blackbox и G-Eval

Главное различие:

    Blackbox = оцениваем финальный ответ
    Golden Path = оцениваем путь агента
    G-Eval = механизм LLM-as-judge в DeepEval

Форматы данных:

  Формат           Что содержит                                      Где нужен
  ---------------- ------------------------------------------------- ---------------------------------
  input + output   задача + финальный ответ                          Blackbox
  Диалог           user/assistant messages                           Blackbox по multi-turn
  Trajectory       последовательность tool calls                     Golden Path по порядку действий
  Trace            tool calls + args + observations + final answer   Golden Path + grounding
  Full trace       trace + spans/timing/errors/metadata/costs        observability/OTEL/debug

Blackbox

Blackbox отвечает на вопрос: хороший ли финальный ответ для пользователя?

Вход:

    {
        "input": user_task,
        "actual_output": agent_final_answer,
        "expected_output": reference_expectation,
        "criteria": evaluation_criteria,
    }

criteria — это критерии оценки / короткая инструкция LLM-судье.

Выход:

    {"score": 1.0, "success": True, "reason": "..."}

С чем имеем дело: финальный текст или диалог. Blackbox не видит tool calls и не доказывает, что агент реально сходил за evidence.

Golden Path

Golden Path отвечает на вопрос: агент прошёл правильный путь?

Вход:

    {
        "actual_trace": [
            {"tool_name": "geocode_location", "arguments": {...}, "observation": {...}},
            {"tool_name": "get_weather_forecast", "arguments": {...}, "observation": {...}},
        ],
        "expected_trace": [
            {"tool_name": "geocode_location"},
            {"tool_name": "get_weather_forecast"},
        ],
    }

Выход:

    {"score": 1.0, "success": True, "reason": "..."}

С чем имеем дело: trajectory/trace, порядок tools, args, observations и иногда final answer. Golden Path ловит missing/extra tools, неправильный порядок и ответ без evidence.

G-Eval

G-Eval не отдельный сценарий. Это способ DeepEval попросить LLM-судью выставить score по критериям.

Для Blackbox G-Eval видит только поля LLMTestCase:

    LLMTestCase(
        input=user_task,
        actual_output=agent_final_answer,
        expected_output=reference_expectation,
    )

Для Golden Path через G-Eval мы сами кладём trace в текст:

    LLMTestCase(
        input=user_task,
        actual_output=serialize_trace_with_observations(agent_trace),
        expected_output="Expected path: geocode_location -> get_weather_forecast",
    )

Коротко:

    Blackbox работает с ответом.
    Golden Path работает с trace/trajectory.
    G-Eval работает с тем, что мы положили в LLMTestCase.

Примеры eval-кода

DeepEval Blackbox

    test_case = LLMTestCase(
        input=case.input,
        actual_output=case.actual_output,
        expected_output=case.expected_output,
    )

    metric = GEval(
        name="Blackbox Quality",
        criteria=case.criteria,
        evaluation_params=[
            SingleTurnParams.INPUT,
            SingleTurnParams.ACTUAL_OUTPUT,
            SingleTurnParams.EXPECTED_OUTPUT,
        ],
        model=judge_model,
        threshold=0.5,
    )

    result = metric.measure(test_case)

DeepEval Golden Path

Deterministic gate:

    test_case = LLMTestCase(
        input=case.input,
        actual_output=case.actual_output,
        tools_called=[
            ToolCall(name=t.name, input_parameters=t.arguments)
            for t in case.actual_tools
        ],
        expected_tools=[
            ToolCall(name=t.name, input_parameters=t.arguments)
            for t in case.expected_tools
        ],
    )

    metric = ToolCorrectnessMetric(
        threshold=1.0,
        should_exact_match=True,
        should_consider_ordering=True,
    )

    result = metric.measure(test_case)

LLM-as-judge по trace:

    trajectory_case = LLMTestCase(
        input=case.input,
        actual_output=serialize_trace_with_observations(case),
        expected_output="Expected path: geocode_location -> get_weather_forecast",
    )

    metric = GEval(
        name="Golden Path Trajectory",
        criteria="Check tool order, missing tools, extra tools and grounding.",
        evaluation_params=[
            SingleTurnParams.INPUT,
            SingleTurnParams.ACTUAL_OUTPUT,
            SingleTurnParams.EXPECTED_OUTPUT,
        ],
        model=judge_model,
        threshold=0.5,
    )

OpenEvals Golden Path, для сравнения

    outputs = to_openai_tool_call_messages(case.actual_tools)
    reference_outputs = to_openai_tool_call_messages(case.expected_tools)

    evaluator = create_trajectory_llm_as_judge(
        judge=judge,
        model=judge_model,
    )

    result = evaluator(
        outputs=outputs,
        reference_outputs=reference_outputs,
    )

Это короче и нативнее, чем DeepEval-вариант. Но разница только в удобстве adapter-а, не в возможности закрыть сценарий.

Можно ли без OpenEvals?

Да. Выбираем DeepEval-only как основной путь.

Что нужно сделать в платформе:

1.  завести нейтральный EvalCase;
2.  для Blackbox маппить его в LLMTestCase + GEval;
3.  для Golden Path маппить trace в ToolCall[];
4.  для Golden Path LLM-as-Judge сериализовать trace в LLMTestCase.actual_output;
5.  нормализовать output всех evals к единому формату платформы.

OpenEvals можно добавить позже только если native trajectory API начнёт реально экономить поддержку. На текущий момент это optional, не обязательная зависимость.

Итог

    Base path: DeepEval-only
    OpenEvals: optional, не обязательная зависимость
    Reason: DeepEval выигрывает по community, гибкости сценариев и tracing/OTEL story
    Trade-off: Golden Path в OpenEvals нативнее, но в DeepEval реализуется без критичной сложности

Минимальный platform contract:

    EvalCase(
        input=user_task,
        actual_output=agent_final_answer,
        expected_output=reference_expectation,
        actual_tools=actual_trace.tool_calls,
        expected_tools=golden_path.tool_calls,
        criteria=evaluation_criteria,
    )
